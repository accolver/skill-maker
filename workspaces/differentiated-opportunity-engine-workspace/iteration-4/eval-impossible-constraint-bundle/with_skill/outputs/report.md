# Differentiation Profile

- Skills and domain experience: Junior full-stack developer; can build simple
  web apps, integrations, CRUD products, and basic AI features.
- Assets and access advantages: No niche audience, no proprietary data, no warm
  buyer access, no special domain access, no budget.
- Distribution and network leverage: None stated beyond public internet
  channels.
- Constraints: Only 2 hours per week; wants `$3000 MRR` in 30 days; refuses
  sales calls, services, content creation, and cold outreach.
- Credibility signals: Can ship software, but no stated domain authority or
  existing user trust.
- Unfair advantages: None currently evident over a random competent developer
  using AI tools.

## Feasibility Verdict

`no_go`

This is an impossible constraint bundle for the requested outcome. The user
wants fast recurring revenue from a cold start while refusing every normal path
to early demand discovery and customer acquisition:

- no audience
- no special access
- no budget
- no sales calls
- no services
- no content creation
- no cold outreach
- only 2 hours per week
- target of `$3000 MRR` within 30 days

That combination leaves no credible route to enough buyer conversations, product
iteration, distribution, or conversion volume. The right answer is not a clever
micro-SaaS idea. The right answer is to change at least one binding constraint.

# Best Bet

The best bet is `No-go for now with a 30-day edge-building plan`.

Why this beats the obvious alternative:

- A pure micro-SaaS-first build loses because the user has no proprietary demand
  signal, no built-in distribution, and no time budget for repeated iteration.
- A generic AI wrapper loses because there is no founder edge and no customer
  acquisition wedge.
- A content-led path loses because the user refuses content creation and has no
  existing audience.
- A service-first wedge would normally be best, but the user explicitly rejects
  services and sales calls.

What must change for a credible business to exist:

The user must change at least one of these:

1. Increase time from `2 hours/week` to at least `5-7 hours/week`.
2. Allow async selling, even if still refusing calls.
3. Allow a narrow service or concierge phase.
4. Lower the target from `$3000 MRR in 30 days` to
   `first $100-$300 revenue in 30-60 days`.
5. Commit to one niche long enough to build a small edge.

If none of those can change, the recommendation is simple: do not try to build
an AI micro-SaaS right now.

# 7-Day Validation Plan

Goal for this week: determine whether there is any realistic wedge without
violating the user's constraints.

1. Pick exactly `3` boring niches with public pain and existing software spend:
   examples include recruiters, property managers, ecommerce operators,
   accountants, or agencies.
2. In each niche, collect `20` public complaints from Reddit, app store reviews,
   G2/Capterra reviews, support forums, or job posts. Total target: `60` pain
   signals.
3. Tag each complaint into a spreadsheet with `problem`, `who has it`,
   `frequency`, `current workaround`, and `would AI materially help`. Keep only
   problems that appear at least `8` times across sources.
4. Identify `10` existing products per surviving problem and note pricing,
   positioning, and review complaints. Total target: `20-30` competitors
   reviewed.
5. Keep only problems where users already pay at least `$20/month` for adjacent
   tools or manual labor.
6. Draft `3` one-sentence product wedges that can be explained in under `15`
   seconds and built as a tiny utility, not a platform.
7. Publish `1` simple landing page or waitlist for the best wedge in a
   built-in-demand channel such as a marketplace listing, relevant directory, or
   niche community post if allowed. Numeric target: at least `25` unique visits
   and `3` email signups in `7` days without paid ads.

Interpretation:

- If the user cannot complete this in `~2 hours`, the time constraint alone is
  disqualifying.
- If no problem survives the frequency and willingness-to-pay filters, there is
  no valid wedge yet.

# Build Spec

Build nothing bigger than a fake-door test plus one tiny prototype.

- Product shape: single-purpose micro-tool for one repetitive workflow, one
  input, one output, one buyer.
- Scope limit: no auth complexity, no team features, no dashboards, no chatbot
  shell, no multi-step agent claims.
- Stack: whatever the user already ships fastest with.
- AI usage: only where it removes manual effort on a narrow task, such as
  classification, extraction, summarization, or rewrite.
- Distribution requirement: must live in a channel with existing buyer intent
  such as a template marketplace, app marketplace, or niche directory. If no
  channel exists, reject the idea.
- Success metric: signal of demand before build depth, not feature completeness.
- Productization trigger: only build beyond MVP if the fake-door test gets
  `>= 3` signups from `<= 50` targeted visits or `>= 1` prepayment.

# Kill Criteria

Stop or pivot if any of these happen:

- By day 7, fewer than `60` public pain signals collected.
- No single problem appears at least `8` times across independent sources.
- No evidence that buyers already spend money on the problem.
- Landing page gets fewer than `25` qualified visits or fewer than `3` signups
  in `7` days.
- The user cannot maintain even `2 focused hours` weekly for `4` straight weeks.
- The user refuses to relax at least one binding constraint after the first
  week.

## Rejected Paths

- Generic AI wrapper: rejected because there is no founder edge, no audience,
  and no distribution wedge.
- Pure content play: rejected because the user refuses content creation and has
  no existing audience.
- Software-first broad micro-SaaS: rejected because discovery, positioning, and
  distribution are too weak under the time and channel constraints.

## Bottom Line

Under the current constraints, the user should not expect `$3000 MRR` from an AI
micro-SaaS in 30 days.

The correct move is:

1. Change at least one constraint.
2. Spend the next 30 days finding a narrow, repeated, paid problem.
3. Only then build the smallest possible utility in a channel with built-in
   demand.

If the user refuses to change any constraint, the honest recommendation is to do
nothing and revisit later.
