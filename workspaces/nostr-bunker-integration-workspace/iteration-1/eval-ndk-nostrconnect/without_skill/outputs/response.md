# NIP-46 Remote Signing with NDK: `nostrconnect://` Flow

## Overview

NIP-46 (Nostr Remote Signing) allows your app to delegate signing operations to
a separate "remote signer" (also called a "bunker"). This keeps the user's
private keys isolated from your application, dramatically reducing the attack
surface.

There are two connection flows in NIP-46:

1. **Bunker flow** (`bunker://`) — The remote signer initiates the connection by
   providing a `bunker://` URI to the client.
2. **NostrConnect flow** (`nostrconnect://`) — The client initiates the
   connection by generating a `nostrconnect://` URI that the user scans with
   their signer app (e.g., via QR code).

This guide focuses on the **NostrConnect flow** — where your app generates the
QR code and waits for the signer to connect.

---

## Key Concepts

### Terminology

| Term                     | Description                                                                                                     |
| ------------------------ | --------------------------------------------------------------------------------------------------------------- |
| **Client**               | Your application — the user-facing app that needs events signed                                                 |
| **Remote Signer**        | The daemon/app that holds the user's keys and responds to signing requests (e.g., Amber, nsecBunker)            |
| **Client Keypair**       | An ephemeral keypair generated by your app, used to encrypt NIP-46 communication. Disposable — delete on logout |
| **Remote Signer Pubkey** | The pubkey used by the remote signer for encrypted communication (may differ from the user's pubkey)            |
| **User Pubkey**          | The actual pubkey representing the user — retrieved via `get_public_key` after connection                       |

### How NostrConnect Works

1. Your app generates a `client-keypair` (ephemeral, for NIP-44 encrypted
   communication)
2. Your app constructs a `nostrconnect://` URI containing the `client-pubkey`,
   relay(s), a secret, and optional metadata
3. The user scans this URI with their signer app (e.g., via QR code)
4. The signer app sends a `connect` response event (kind `24133`) to the client
   via the specified relays
5. Your app discovers the `remote-signer-pubkey` from the connect response
6. Your app calls `get_public_key` to learn the actual `user-pubkey`
7. From this point, your app can send signing requests

## Implementation with NDK

### 1. Setup NDK

```typescript
import NDK, { NDKNip46Signer } from "@nostr-dev-kit/ndk";

// Initialize NDK with relays your app uses for general Nostr operations
const ndk = new NDK({
  explicitRelayUrls: [
    "wss://relay.damus.io",
    "wss://relay.primal.net",
    "wss://nos.lol",
  ],
});

await ndk.connect();
```

### 2. Create the NostrConnect Signer and Generate the URI

NDK provides `NDKNip46Signer.nostrconnect()` as a static factory method
specifically for the client-initiated flow:

```typescript
// Choose a relay for NIP-46 communication between your app and the signer
const nip46Relay = "wss://relay.primal.net";

// If you have a previously saved local nsec (for session restoration), provide it.
// Otherwise pass undefined and NDK will generate a new ephemeral keypair.
const savedLocalNsec: string | undefined = loadFromStorage("nip46_local_nsec");

// Create the signer
const signer = NDKNip46Signer.nostrconnect(ndk, nip46Relay, savedLocalNsec, {
  name: "My Nostr App",
  icon: "https://myapp.example.com/icon.png",
  perms: ["sign_event:1", "sign_event:30023", "nip44_encrypt", "nip44_decrypt"],
});

// The nostrConnectUri is now available — this is what you display as a QR code
const connectUri = signer.nostrConnectUri;
console.log("NostrConnect URI:", connectUri);
```

The generated URI looks like:

```
nostrconnect://83f3b2ae6aa368e8275397b9c26cf550101d63ebaab900d19dd4a4429f5ad8f5?relay=wss%3A%2F%2Frelay.primal.net&perms=sign_event%3A1%2Csign_event%3A30023%2Cnip44_encrypt%2Cnip44_decrypt&name=My+Nostr+App&secret=a1b2c3d4
```

### 3. Display the QR Code

Use any QR code library to render the URI. Here's an example with `qrcode` (for
Node/browser) or a React component:

#### Plain JavaScript / Node

```typescript
import QRCode from "qrcode";

// Generate QR code as a data URL for an <img> tag
const qrDataUrl = await QRCode.toDataURL(connectUri, {
  width: 300,
  margin: 2,
  color: {
    dark: "#000000",
    light: "#ffffff",
  },
});

// In a browser context:
const img = document.createElement("img");
img.src = qrDataUrl;
document.getElementById("qr-container")?.appendChild(img);
```

#### React Component

```tsx
import { useEffect, useState } from "react";
import QRCode from "qrcode";

function NostrConnectQR({ uri }: { uri: string }) {
  const [qrSrc, setQrSrc] = useState<string>("");

  useEffect(() => {
    QRCode.toDataURL(uri, { width: 300, margin: 2 }).then(setQrSrc);
  }, [uri]);

  return (
    <div>
      <h3>Scan with your Nostr signer app</h3>
      {qrSrc
        ? <img src={qrSrc} alt="NostrConnect QR Code" />
        : <p>Generating...</p>}
      <p style={{ fontSize: "0.8em", wordBreak: "break-all" }}>{uri}</p>
    </div>
  );
}
```

### 4. Wait for the Signer to Connect

After displaying the QR code, call `blockUntilReady()` which returns a promise
that resolves once the remote signer has connected and the user's public key has
been retrieved:

```typescript
try {
  // This blocks until the remote signer scans the QR and responds
  const user = await signer.blockUntilReady();

  console.log("Connected! User npub:", user.npub);

  // IMPORTANT: Save the local signer's nsec for session restoration
  // This allows reconnecting without scanning the QR code again
  const localNsec = signer.localSigner.nsec;
  saveToStorage("nip46_local_nsec", localNsec);

  // Set the signer on NDK so all subsequent operations use it
  ndk.signer = signer;
} catch (error) {
  console.error("Failed to connect to remote signer:", error);
}
```

### 5. Use the Signer for Signing Events

Once connected, NDK uses the remote signer transparently. You publish events the
same way you would with any other signer:

```typescript
import { NDKEvent } from "@nostr-dev-kit/ndk";

// Create and publish a kind:1 note — NDK handles the remote signing
const event = new NDKEvent(ndk);
event.kind = 1;
event.content = "Hello from my app with remote signing!";
event.tags = [];

// This sends a sign_event request to the remote signer via NIP-46,
// waits for the signed event back, then publishes it
await event.publish();

console.log("Published event:", event.id);
```

### 6. Encryption via Remote Signer

If you requested `nip44_encrypt` / `nip44_decrypt` permissions, you can use the
signer for encryption too:

```typescript
// NIP-44 encrypt
const ciphertext = await signer.encrypt(
  recipientUser, // NDKUser instance
  "Secret message",
);

// NIP-44 decrypt
const plaintext = await signer.decrypt(
  senderUser, // NDKUser instance
  ciphertext,
);
```

## Complete Integration Example

Here's a full end-to-end example combining all the pieces:

```typescript
import NDK, { NDKEvent, NDKNip46Signer } from "@nostr-dev-kit/ndk";

async function setupRemoteSigning() {
  // 1. Initialize NDK
  const ndk = new NDK({
    explicitRelayUrls: [
      "wss://relay.damus.io",
      "wss://relay.primal.net",
      "wss://nos.lol",
    ],
  });
  await ndk.connect();

  // 2. Check for existing session
  const savedNsec = localStorage.getItem("nip46_local_nsec") ?? undefined;

  // 3. Create NostrConnect signer
  const signer = NDKNip46Signer.nostrconnect(
    ndk,
    "wss://relay.primal.net",
    savedNsec,
    {
      name: "My Nostr App",
      icon: "https://myapp.example.com/icon.png",
      perms: ["sign_event:1", "sign_event:30023"],
    },
  );

  // 4. Display QR code to user
  displayQRCode(signer.nostrConnectUri);
  showStatus("Waiting for signer connection...");

  // 5. Wait for connection
  try {
    const user = await signer.blockUntilReady();

    // 6. Save session for future use
    localStorage.setItem("nip46_local_nsec", signer.localSigner.nsec);

    // 7. Set signer on NDK
    ndk.signer = signer;

    hideQRCode();
    showStatus(`Connected as ${user.npub}`);

    // 8. Now you can sign events
    await publishTestNote(ndk);
  } catch (err) {
    showStatus("Connection failed: " + err);
  }
}

async function publishTestNote(ndk: NDK) {
  const event = new NDKEvent(ndk);
  event.kind = 1;
  event.content = "Testing remote signing with NIP-46!";
  await event.publish();
  console.log("Published:", event.id);
}
```

## Session Restoration (Bunker Flow)

If the user has previously connected and you saved the `localNsec` and the
bunker connection string, you can restore the session without showing a QR code:

```typescript
const savedBunkerUri = localStorage.getItem("bunker_uri");
const savedLocalNsec = localStorage.getItem("nip46_local_nsec");

if (savedBunkerUri && savedLocalNsec) {
  // Restore existing session — no QR code needed
  const signer = NDKNip46Signer.bunker(ndk, savedBunkerUri, savedLocalNsec);
  const user = await signer.blockUntilReady();
  ndk.signer = signer;
  console.log("Session restored for", user.npub);
} else {
  // First-time connection — show QR code
  // ... (nostrconnect flow as above)
}
```

## How `switch_relays` Works and Why It Matters

### What It Does

`switch_relays` is a NIP-46 method that allows the **remote signer** to tell the
client to change which relays are used for their NIP-46 communication channel.
The client sends a `switch_relays` request, and the remote signer responds with
either:

- An array of relay URLs (the new relays to use), or
- `null` (meaning no change is needed)

### The Protocol Flow

```
Client                          Remote Signer
  |                                  |
  |--- switch_relays request ------->|
  |                                  |
  |<-- ["wss://new1", "wss://new2"] -|
  |                                  |
  | (client disconnects from old     |
  |  relays, connects to new ones)   |
  |                                  |
  |--- future requests on new relays>|
```

### Why It Matters

1. **Relay Sovereignty**: The remote signer should always control which relays
   are used for the NIP-46 communication channel. When the client initiates via
   `nostrconnect://`, it picks relays that may be completely foreign to the
   signer's preferences. `switch_relays` lets the signer redirect to its
   preferred infrastructure.

2. **Relay Lifecycle**: Relays go offline, change policies, or become unreliable
   over time. `switch_relays` allows the signer to migrate the communication
   channel to new relays without breaking the connection or requiring the user
   to re-scan a QR code.

3. **Performance & Reliability**: The signer knows which relays it has the best
   connectivity to. By switching to its preferred relays, message delivery
   becomes more reliable and latency decreases.

4. **Privacy**: The signer may prefer relays with specific privacy properties
   (no logging, Tor support, etc.). `switch_relays` lets it enforce these
   preferences regardless of what the client initially chose.

### When to Call It

Per the NIP-46 spec:

> Compliant clients should send a `switch_relays` request **immediately upon
> establishing a connection** (always, or at reasonable intervals).

This means your client should:

1. Call `switch_relays` right after the `connect` handshake completes
2. Optionally call it periodically during long-lived sessions
3. Immediately update its local relay state when it receives a non-null response
4. Send all subsequent NIP-46 requests on the new relays

### Implementation Note

NDK handles `switch_relays` internally as part of its NIP-46 signer
implementation. When you use `NDKNip46Signer`, the relay switching is managed
for you. However, understanding the mechanism is important for:

- Debugging connection issues (if the signer switches to a relay your app can't
  reach)
- Understanding why NIP-46 communication might move to different relays than you
  initially specified
- Building custom NIP-46 implementations outside of NDK

## NIP-46 Protocol Details

### Event Kind

All NIP-46 communication uses **kind `24133`** events. Both requests and
responses use this same kind.

### Encryption

All message content is encrypted using **NIP-44** encryption. The client
encrypts to the `remote-signer-pubkey`, and the signer encrypts to the
`client-pubkey`.

### Message Format

Requests follow a JSON-RPC-like structure:

```json
{
  "id": "random-request-id",
  "method": "sign_event",
  "params": [
    "{\"kind\":1,\"content\":\"Hello\",\"tags\":[],\"created_at\":1714078911}"
  ]
}
```

Responses:

```json
{
  "id": "random-request-id",
  "result": "<signed-event-json>",
  "error": null
}
```

### Available Methods

| Method           | Purpose                       |
| ---------------- | ----------------------------- |
| `connect`        | Establish the connection      |
| `get_public_key` | Get the user's actual pubkey  |
| `sign_event`     | Sign a Nostr event            |
| `nip04_encrypt`  | Encrypt using NIP-04          |
| `nip04_decrypt`  | Decrypt using NIP-04          |
| `nip44_encrypt`  | Encrypt using NIP-44          |
| `nip44_decrypt`  | Decrypt using NIP-44          |
| `ping`           | Health check (returns "pong") |
| `switch_relays`  | Request updated relay list    |

### Auth Challenges

The remote signer may respond with an auth challenge if it needs the user to
authenticate through another mechanism (e.g., a web page). The response will
have `result: "auth_url"` and the `error` field will contain a URL to display to
the user. Your app should open this URL and continue listening for the actual
response on the same request ID.

## Security Considerations

1. **Always validate the secret**: When using `nostrconnect://`, include a
   `secret` parameter and validate that the connect response returns the same
   secret. This prevents connection spoofing.

2. **Store the local nsec securely**: The ephemeral client keypair
   (`localSigner.nsec`) should be stored securely (encrypted storage, keychain,
   etc.) — not in plain localStorage in production.

3. **Request minimal permissions**: Only request the permissions your app
   actually needs in the `perms` field.

4. **Handle disconnections**: Remote signers can go offline. Implement retry
   logic and inform the user when the signer is unreachable.

5. **Delete keys on logout**: When the user logs out, delete the stored local
   nsec to prevent session reuse.
