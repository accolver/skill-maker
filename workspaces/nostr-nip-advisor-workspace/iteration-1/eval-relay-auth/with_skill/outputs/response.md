# Implementing Relay Authentication

You need two NIPs: **NIP-42** for the authentication protocol itself, and
**NIP-11** for the relay information document that tells clients what
authentication is required.

---

## NIP-42 — Client Authentication

**NIP-42** is the primary NIP for relay authentication. It defines a
challenge-response flow using **kind:22242** events.

### Authentication Flow

```
1. Relay → Client:  ["AUTH", "<challenge-string>"]
2. Client → Relay:  ["AUTH", <signed-kind-22242-event>]
3. Relay → Client:  ["OK", "<event-id>", true, ""]
```

Here's each step in detail:

#### Step 1: Relay Sends Challenge

When a client connects (or when auth is needed for a specific action), the relay
sends an AUTH message with a challenge string:

```json
["AUTH", "<random-challenge-string>"]
```

The challenge is a random string generated by the relay to prevent replay
attacks.

#### Step 2: Client Signs kind:22242

The client creates and signs a kind:22242 authentication event with `relay` and
`challenge` tags:

```jsonc
{
  "kind": 22242,
  "tags": [
    ["relay", "wss://relay.example.com/"],
    ["challenge", "<challenge-string-from-relay>"]
  ],
  "content": "",
  "created_at": "<current-timestamp>",
  "pubkey": "<client-pubkey>",
  "sig": "<signature>"
}
```

The client then sends this signed event back to the relay:

```json
["AUTH", <signed-kind-22242-event>]
```

#### Step 3: Relay Validates and Responds

The relay validates:

- The `relay` tag matches its own URL
- The `challenge` tag matches the challenge it sent
- The event signature is valid
- The `created_at` is recent (within a reasonable time window)

On success, the relay responds with:

```json
["OK", "<event-id>", true, ""]
```

### When Auth is Required

Auth can be required for specific actions rather than globally. Common patterns:

- **Reading kind:4 DMs** — relay requires auth to serve encrypted DM events, so
  only the intended recipient can fetch them
- **Publishing events** — relay restricts who can write to it (paid relay,
  community relay, etc.)
- **Reading restricted content** — relay-specific access control

When a client tries to perform an action that requires auth, the relay responds
with a rejection that includes the `auth-required:` prefix:

For **EVENT** submissions:

```json
["OK", "<event-id>", false, "auth-required: you must authenticate to publish"]
```

For **REQ** subscriptions:

```json
["CLOSED", "<sub-id>", "auth-required: you must authenticate to read DMs"]
```

The client should then complete the AUTH flow and retry the action.

## NIP-11 — Relay Information Document

**NIP-11** defines the relay information document, which tells clients about the
relay's capabilities and requirements before they connect via WebSocket.

### Fetching the Info Document

Clients fetch it via HTTP GET to the relay URL with the
`Accept: application/nostr+json` header:

```
GET https://relay.example.com/
Accept: application/nostr+json
```

### Advertising Auth Requirements

The relay info document uses the `limitation` field to advertise that
authentication is required:

```jsonc
{
  "name": "My Authenticated Relay",
  "description": "A relay that requires authentication",
  "pubkey": "<admin-pubkey>",
  "supported_nips": [1, 11, 42],
  "software": "https://github.com/example/relay",
  "version": "1.0.0",
  "limitation": {
    "auth_required": true,
    "payment_required": false,
    "max_message_length": 16384,
    "max_subscriptions": 20,
    "max_event_tags": 100
  }
}
```

The key field is `"auth_required": true` in the `limitation` object. This tells
clients that they will need to complete the AUTH flow before they can interact
with the relay.

### Why NIP-11 Matters for Auth

Without NIP-11, clients have no way to know authentication is required until
they try to publish or subscribe and get rejected. By advertising
`auth_required` in the relay info document:

- Clients can prompt users to authenticate proactively
- Clients can display the relay's auth requirements in relay management UIs
- Clients can skip relays that require auth if the user hasn't configured
  credentials

## Implementation Checklist

### Relay Side

1. Generate a unique challenge string per connection
2. Send `["AUTH", "<challenge>"]` on connect (or on demand)
3. Validate kind:22242 events (relay tag, challenge tag, signature, timestamp)
4. Respond with `["OK", ...]` on success
5. Use `auth-required:` prefix in `OK` and `CLOSED` messages when auth is needed
6. Serve NIP-11 info document with `auth_required` in `limitation`

### Client Side

1. Fetch NIP-11 info document before connecting
2. Handle `["AUTH", "<challenge>"]` messages
3. Sign kind:22242 with correct relay and challenge tags
4. Send `["AUTH", <signed-event>]`
5. Handle `auth-required:` rejections by re-authenticating
6. Retry the failed action after successful auth
