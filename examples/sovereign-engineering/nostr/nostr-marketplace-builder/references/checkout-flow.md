# Checkout Flow Reference

Complete checkout flow for NIP-15 marketplace transactions using NIP-04
encrypted direct messages.

## Overview

Checkout is a 3-step message exchange between customer and merchant, sent as
JSON strings inside NIP-04 encrypted DMs. Each message has a `type` field that
determines its role in the flow.

```
Customer                          Merchant
   |                                 |
   |--- Order (type:0) ------------->|
   |                                 |
   |<-- Payment Request (type:1) ----|
   |                                 |
   |    [Customer pays externally]   |
   |                                 |
   |<-- Status Update (type:2) ------|
   |    (paid: true)                 |
   |                                 |
   |<-- Status Update (type:2) ------|
   |    (shipped: true)              |
```

## Message Types

| Type | Direction           | Purpose                              |
| ---- | ------------------- | ------------------------------------ |
| 0    | Customer → Merchant | New order with items and shipping    |
| 1    | Merchant → Customer | Payment request with payment options |
| 2    | Merchant → Customer | Order status update (paid/shipped)   |

---

## Type 0: Customer Order

Sent by the customer to initiate a purchase.

```json
{
  "id": "<string, order ID generated by customer>",
  "type": 0,
  "name": "<string, optional, customer name>",
  "address": "<string, optional, shipping address for physical goods>",
  "message": "<string, optional, message to merchant>",
  "contact": {
    "nostr": "<string, 32-byte hex pubkey>",
    "phone": "<string, optional>",
    "email": "<string, optional>"
  },
  "items": [
    {
      "product_id": "<string, product id from kind:30018>",
      "quantity": "<int, number of units>"
    }
  ],
  "shipping_id": "<string, chosen shipping zone id from stall>"
}
```

**Field Details:**

| Field              | Type   | Required | Notes                                          |
| ------------------ | ------ | -------- | ---------------------------------------------- |
| id                 | string | Yes      | Customer-generated order ID                    |
| type               | int    | Yes      | Must be `0`                                    |
| name               | string | No       | Customer display name                          |
| address            | string | No       | Required for physical goods                    |
| message            | string | No       | Free-text message to merchant                  |
| contact.nostr      | string | Yes*     | Customer's Nostr pubkey                        |
| contact.phone      | string | No       | Phone contact                                  |
| contact.email      | string | No       | Email contact                                  |
| items              | array  | Yes      | At least one item                              |
| items[].product_id | string | Yes      | Must match a product's `id`                    |
| items[].quantity   | int    | Yes      | Number of units to purchase                    |
| shipping_id        | string | Yes      | Must match a shipping zone `id` from the stall |

**Validation Rules:**

- `shipping_id` must match one of the shipping zones defined in the merchant's
  stall (kind:30017)
- All `product_id` values must reference products (kind:30018) belonging to the
  same stall
- `quantity` must not exceed the product's available `quantity` (unless null)

## Type 1: Payment Request

Sent by the merchant after receiving and validating the order.

```json
{
  "id": "<string, same order ID from type:0>",
  "type": 1,
  "message": "<string, optional, message to customer>",
  "payment_options": [
    {
      "type": "<string, payment type>",
      "link": "<string, payment destination>"
    }
  ]
}
```

**Payment Option Types:**

| Type    | Description                           | Link Format               |
| ------- | ------------------------------------- | ------------------------- |
| `url`   | Payment page (Stripe, PayPal, BTCPay) | HTTPS URL                 |
| `btc`   | On-chain Bitcoin address              | Bitcoin address (bc1q...) |
| `ln`    | Lightning Network invoice             | BOLT11 invoice (lnbc...)  |
| `lnurl` | LNURL-pay endpoint                    | LNURL string              |

**Rules:**

- The `id` MUST match the order ID from the customer's type:0 message
- At least one payment option should be provided
- Multiple payment options give the customer flexibility

---

## Type 2: Order Status Update

Sent by the merchant to update the customer on order progress. Can be sent
multiple times as the order progresses.

```json
{
  "id": "<string, same order ID>",
  "type": 2,
  "message": "<string, status message to customer>",
  "paid": "<bool, whether payment has been received>",
  "shipped": "<bool, whether the order has been shipped>"
}
```

**Typical Status Progression:**

1. `{"paid": false, "shipped": false}` — Order received, awaiting payment
2. `{"paid": true, "shipped": false}` — Payment confirmed, preparing shipment
3. `{"paid": true, "shipped": true}` — Order shipped

**Rules:**

- The `id` MUST match the original order ID
- `paid` and `shipped` are booleans, not strings
- Multiple type:2 messages can be sent as status changes

## Shipping Cost Calculation at Checkout

When processing a type:0 order, calculate total cost:

```
For each item in order:
  item_cost = product.price × item.quantity
  item_shipping = product.shipping[shipping_id].cost × item.quantity (if exists, else 0)

subtotal = SUM(item_cost for all items)
shipping = stall.shipping[shipping_id].cost + SUM(item_shipping for all items)
total = subtotal + shipping
```

**Example:**

Stall shipping zone "us-domestic" base cost: $5.99 Product A: $29.99, shipping
$2.00/unit, ordering 3 Product B: $49.99, shipping $0 (no extra), ordering 1

```
subtotal = (29.99 × 3) + (49.99 × 1) = $139.96
shipping = 5.99 + (2.00 × 3) + (0 × 1) = $11.99
total = $151.95
```

## Customer Support

After checkout, customer support communication happens over the same channel
(NIP-04 DMs) or whatever contact method was specified in the order's `contact`
field. There is no specific message type for support — it's free-form
communication between merchant and customer.
